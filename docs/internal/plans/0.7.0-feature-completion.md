# 0.7.0 Feature Completion Plan

**Created: 2026-03-01**
**Target: 0.7.0 release (finalize before 0.8.0 begins)**

This plan captures the concrete features required to close out 0.7.0 as a
complete release before moving to 0.8.0 (SDL3 migration / plugin architecture).

---

## Guiding Principles

1. **Close stub gaps** — every `UnimplementedError` in a shipped editor should
   either be implemented or explicitly deferred with a comment explaining why.
2. **ZScream parity** — the roadmap's Track A parity table is the benchmark.
3. **Test coverage** — new features must include tests. Editors without any
   test file get at least a basic smoke test.
4. **No new subsystems** — 0.7.0 is about finishing, not starting.

## Dungeon Draw Routine Dedupe Plan (post-0.7.0 hardening)

Objective: keep a single source of truth for dungeon object drawing behavior so
usdasm parity fixes cannot drift between registry and runtime paths.

- [x] Phase 1: corner routine unification
  - Runtime dispatch for routines `19`, `35`, `36`, `37` now routes through
    `ObjectDrawer::DrawUsingRegistryRoutine`.
  - Removed duplicated legacy corner implementations from `ObjectDrawer`.
  - Coverage guardrails: `ObjectDrawerRegistryReplayTest`,
    `DrawRoutineBothBGMetadataTest`, `ObjectGeometryTest`.
- [ ] Phase 2: migrate remaining duplicated routine families
  - Migrate routine families that exist in both registry modules and
    `ObjectDrawer` methods (start with rightwards/downwards families that already
    have registry routines).
  - Progress:
    - ✅ Diagonal family migrated (`5`, `6`, `17`, `18`)
    - ✅ Core downwards family migrated (`7`-`15`)
  - After each family migration, remove dead `ObjectDrawer` method declarations
    and implementations.
- [ ] Phase 3: registry-first dispatcher policy
  - Add a dispatcher rule: if `DrawRoutineRegistry::GetRoutineInfo(id)` exists,
    default to registry path unless explicitly marked legacy-only.
  - Keep a temporary opt-out list for routines that still depend on bespoke
    state hooks.
- [ ] Phase 4: parity and regression gate expansion
  - Add coverage that compares registry replay bounds/trace footprints against
    `ObjectGeometry` for both subtype-2 and subtype-3 corner/canvas objects.
  - Add a focused regression for custom corner alias objects (`0x100..0x103`)
    to ensure custom payload rendering remains stable while deduping.
- [ ] Phase 5: docs + ownership cleanup
  - Document which dungeon routines are registry-owned vs legacy-owned in
    `src/zelda3/dungeon/README.md`.
  - Add contributor rule: new routine behavior changes must land in registry
    first, runtime second.
