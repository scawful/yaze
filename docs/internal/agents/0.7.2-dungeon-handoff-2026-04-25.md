  exercises the in-memory → ROM round trip.
- **Cite routine bodies, not the audit** — the
  `zelda3-hacking-expert` audits are useful for cataloging gaps but
  have been wrong on specifics multiple times this session
  (transcription typos past entry 6, off-by-one on count semantics,
  inflated cluster IDs). Always re-derive counts and mappings from
  yaze's own routine bodies and registry.
- **No production count changes without a proven under-fetch.**
  Tightening over-fetches (8 → 4 etc.) is a sharpening exercise the
  owner has explicitly deferred until correctness gaps are clearly
  proven by routine bodies. Tests pinning the routine-mapping
  contract are fine; production count edits to fix mere over-fetch
  are out of scope unless explicitly green-lit.
- **Loaded/dirty contract** for chest/pot/block save:
  `SetLoaded(true)` does NOT cascade into per-table `*_loaded_`
  flags. Tests that mutate `GetChests()/GetPotItems()/tile_objects_`
  directly must call the corresponding `Mark*Dirty()` to make
  writers serialize the edit. Otherwise the writer treats the room
  as header-only and preserves vanilla bytes. Reference:
  `memory/feedback_dungeon_save_loaded_dirty_contract.md`.

### Routine-body-proven under-fetch: Turtle Rock pipes (`0xFBA-0xFBD`)

**Severity: Waterfall47/48 tier.** Discovered during in-rotation
review on 2026-04-25 by reading the routine bodies directly:

- `0xFBA`, `0xFBB` → routine 102 (`DrawVerticalTurtleRockPipe` at
  `src/zelda3/dungeon/draw_routines/special_routines.cc:430-437`):
  two stacked 4x3 sections, reads tiles 0..11 then 12..23 → **24 tiles**.
- `0xFBC`, `0xFBD` → routine 103 (`DrawHorizontalTurtleRockPipe` at
  `special_routines.cc:439-441`): `DrawNx4(columns=6, start=0)` →
  `DrawColumnMajor(6, 4, ...)` → **24 tiles**.
- Parser path: `GetSubtype3TileCount` at
  `src/zelda3/dungeon/object_parser.cc:420-506` has no special case
  for `0xFBA-0xFBD`, falls through to the default 8.
- Result: `TileAtWrapped` substitutes tiles 0..7 cyclically for
  indices 8..23 — visible mis-tiling on Turtle Rock pipe sections,
  same failure mode as Waterfall47/48 before `e9938002`.

This is _not_ "needs ROM replay to know" — the count gap is
provable from yaze's own routine bodies. ROM replay confirms; it
does not discover. **Fix is one of the highest-priority dungeon
correctness items remaining for 0.7.2.**

The existing `draw_routine_mapping_test.cc:534-539` only pins
routing (`0xFBA → kVerticalTurtleRockPipe`); no test pins the
parser tile count, so a one-line edit to `GetSubtype3TileCount`
would currently pass without a regression test catching the
correctness improvement. The next session must ship the production
count change with a routine-body-anchored regression test (parse
`0xFBA`, expect `tiles.size() >= 24`; same for `0xFBC`).
